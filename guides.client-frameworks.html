Wiring into client frameworks

Brief

Attach a compiled rule to the validation lifecycle of ASP.NET, jQuery, Angular, React or Vue.

Details

By the time a rule reaches the page, the library has done its part: you have the runtime stub and a compiled rule. Wrapped as a function (see the walkthrough and model rules), that rule is just a boolean:

  • a single-field rule takes the field value - ageOk(value); or

  • a whole-model rule reads the model - passwordsMatch() via reference(), or rule(model) when you compile it for object access.

Wiring it into any framework is therefore the same two steps: call the boolean at the right moment, and map false to an error message. What differs per framework is only where that moment is and how errors are surfaced.

The named rule function (ageOk, passwordsMatch) gets onto the page one of two ways, both from the walkthrough: a server-rendered page embeds it inline (the server already holds the compiler), while a single-page app serves it as a script or bundles it. The ASP.NET section embeds; the rest assume the function is already loaded.

Where does the rule read its inputs?

One choice decides the rest. A DOM-based stack (server-rendered pages, jQuery) keeps the values in form inputs, so a whole-model rule's reference() can read them straight from the DOM. A component stack (Angular, React, Vue) keeps state in the component, not the DOM - so there, compile whole-model rules for object access (value.Field) and call rule(model) with the component's own state. Single-field rules are the same everywhere: hand the field value to rule(value).

ASP.NET

The server is already rendering the page and already holds the compiler, so embed the rule inline rather than fetching it separately: emit it as a function with Razor, right where it is used (the stub goes in the layout, as in the walkthrough). Then call it from a CustomValidator's client function - args.Value is the field, args.IsValid is the verdict:

Embedding the rule and calling it Hide

@* AgeRule is the C# lambda x => x > 0 && x < 100 *@
<asp:TextBox id="Age" runat="server" />
<asp:CustomValidator runat="server" ControlToValidate="Age"
     ClientValidationFunction="validateAge" ErrorMessage="Enter a value between 1 and 99." />
 
<script>
    function ageOk(x) { return @Html.Raw(new ExpressionCompiler(AgeRule).JavaScriptExpression); }
 
    function validateAge(sender, args) {
        args.IsValid = ageOk(args.Value);
    }
</script>

For a whole-model rule, emit it the same way (compiled for reference(), as a no-argument function), attach the validator to any field, and call it - reference() reads the rest of the form.

jQuery

Register the rule as a custom method, then name it in the field's rules. this.optional(element) keeps an empty field from failing (let a required rule own that):

A jQuery Validate method Show

A jQuery Validate method Hide

$.validator.addMethod("ageOk", function (value, element) {
    return this.optional(element) || ageOk(value);
}, "Enter a value between 1 and 99.");
 
$("#signup").validate({
    rules: { age: { ageOk: true } }
});

A whole-model rule (such as the password match) is a method on the confirmation field that ignores its argument and calls the no-argument rule.

Angular

In reactive forms a validator is a function returning null when valid or an error object when not. Put a single-field rule on the control; put a whole-model rule on the FormGroup so it sees every field:

Reactive-form validators Show

Reactive-form validators Hide

// single field
function ageOkValidator(control) {
    return ageOk(control.value) ? null : { ageOk: true };
}
 
// whole model - compiled for object access, fed the form's value
function passwordsMatchValidator(group) {
    return rule(group.value) ? null : { passwordsMatch: true };
}
 
this.form = new FormGroup({
    password: new FormControl(""),
    confirm:  new FormControl(""),
}, { validators: passwordsMatchValidator });

React

React has no built-in validation, so call the rule wherever you check the form - here a controlled component derives validity on each render:

A controlled component Show

A controlled component Hide

const [age, setAge] = useState("");
const valid = ageOk(age);
 
return (
    <>
        <input value={age} onChange={e => setAge(e.target.value)} />
        {!valid && <span className="error">Enter a value between 1 and 99.</span>}
    </>
);

With a form library the rule drops into its hook - for react-hook-form, register("age", { validate: v => ageOk(v) || "Enter a value between 1 and 99." }). A whole-model rule (compiled for object access) takes the form's values object: rule(getValues()).

Vue

In Vue derive validity from state with a computed value:

Computed validity Hide

const age = ref("");
const ageValid = computed(() => ageOk(age.value));
 
// whole model - pass the reactive state object (compiled for object access)
const formValid = computed(() => rule(form));

With VeeValidate, the rule becomes a custom rule: defineRule("ageOk", ageOk).
