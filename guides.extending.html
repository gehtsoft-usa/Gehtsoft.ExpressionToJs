Extending the translator for your own types

Brief

Map a domain method, constant type or property the built-ins do not cover, without subclassing.

Details

Real models reference types the built-ins don't know: a domain helper, a value struct, a Guid. When a rule uses one, the compiler throws at compile time rather than guess - so a gap surfaces on the server, not as wrong JavaScript in the browser. You close the gap by teaching the compiler the mapping, through one of three registries on the compiler instance.

Register before reading JavaScriptExpression; registrations are consulted ahead of the built-ins, so they can also deliberately shadow a default. The three scenarios below use one order-form model:

class Order
{

Only constructs that read a parameter (o.something) reach the client; a subtree with no parameter is computed on the server and emitted as a literal. So these scenarios are all about a model value the browser has to work with.

Add a method for a type

When: your rule calls a domain helper on a model value - a server-side check with no JavaScript twin, such as Coupon.IsValid(o.CouponCode):

The C# helper (server side) Show

The C# helper (server side) Hide

public static class Coupon
{
    public static bool IsValid(string code) =>
        code != null && Regex.IsMatch(code, "^[A-Z]{4}-\\d{4}$");
}

Map it to a JavaScript function with MapMethod; in the template, $0, $1, ... are the call's arguments (and $obj the instance, for an instance method):

Map the call to a JS function Show

Map the call to a JS function Hide

var compiler = new ExpressionCompiler(rule);
compiler.Methods.MapMethod(typeof(Coupon), nameof(Coupon.IsValid), "jsv_coupon_ok($0)");
 
// o => Coupon.IsValid(o.CouponCode)
//   -> jsv_coupon_ok(o.CouponCode)

The JavaScript side: the emitted code calls jsv_coupon_ok, so define it on the page - the same check in JavaScript, next to the runtime stub:

Ship the matching function Show

Ship the matching function Hide

function jsv_coupon_ok(code) {
    return code != null && /^[A-Z]{4}-\d{4}$/.test(code);
}

Add a property read for a type

When: a rule reads a property the built-ins don't translate, so the value must be computed on the client. DateTime.DayOfYear is a good example: a JavaScript Date has no such property, yet by default the compiler emits the access verbatim - silently wrong:

The default - verbatim, and wrong for a JS Date Show

The default - verbatim, and wrong for a JS Date Hide

// o => o.PlacedOn.DayOfYear <= 31
//   -> jsv_lessorequal(o.PlacedOn.DayOfYear, 31)   // a JS Date has no .DayOfYear

Map the property to a real client computation with MapMember; $obj is the emitted target the property is read from:

Map the property to a helper Show

Map the property to a helper Hide

compiler.Members.MapMember(typeof(DateTime), nameof(DateTime.DayOfYear), "jsv_dayofyear($obj)");
 
// o => o.PlacedOn.DayOfYear <= 31
//   -> jsv_lessorequal(jsv_dayofyear(o.PlacedOn), 31)

The JavaScript side: ship the jsv_dayofyear helper that computes it from a Date:

Ship the matching function Show

Ship the matching function Hide

function jsv_dayofyear(d) {
    var start = new Date(d.getFullYear(), 0, 0);
    return Math.floor((d - start) / 86400000);
}

Map an enum to the client

When: the client carries an enum by name rather than by number. By default an enum constant emits its underlying numeric value (so OrderStatus.Shipped becomes 1); MapConstant maps the enum type to a different literal - here its name:

Render the enum by name Show

Render the enum by name Hide

compiler.Methods.MapMethod(typeof(OrderRules), nameof(OrderRules.StatusIs), "jsv_status_is($0, $1)");
compiler.Constants.MapConstant<OrderStatus>(s => "'" + s + "'");
 
// o => OrderRules.StatusIs(o.Status, OrderStatus.Shipped)
//   default       -> jsv_status_is(o.Status, 1)
//   with the map  -> jsv_status_is(o.Status, 'Shipped')

The JavaScript side: a data contract - the client must carry Status as the matching string. One caveat: this fires where the enum stays enum-typed (a method argument, as above). A direct o.Status == OrderStatus.Shipped is lifted to an int comparison by the C# compiler, so the constant is already a number and the map does not apply there.

When a template is not enough

A template is a fixed string with $0/$obj holes. When the emitted form has to vary - on argument types, arity, or the value of an argument - implement the matching interface instead and register it with AddTranslator: IMethodCallTranslator, IConstantTranslator or IMemberTranslator. Each receives the node and an IExpressionEmitContext (to recurse into sub-expressions with Emit, and to escape strings/regex), and returns whether it handled the node. These are the very interfaces the built-ins are written against.

For example, suppose Coupon.IsValid gains a strict flag that should select a different JavaScript function - a choice a fixed template can't make. The translator reads the constant argument and emits accordingly:

A method translator that branches on an argument Show

A method translator that branches on an argument Hide

sealed class CouponTranslator : IMethodCallTranslator
{
    public bool TryTranslate(MethodCallExpression call, IExpressionEmitContext ctx, out string js)
    {
        if (call.Method.DeclaringType == typeof(Coupon) && call.Method.Name == "IsValid"
            && call.Arguments.Count == 2)
        {
            string code = ctx.Emit(call.Arguments[0]);
            bool strict = (bool)((ConstantExpression)call.Arguments[1]).Value;
            js = strict ? $"jsv_coupon_strict({code})" : $"jsv_coupon_ok({code})";
            return true;
        }
        js = null;
        return false;   // not mine - let another translator or the built-ins handle it
    }
}
 
compiler.Methods.AddTranslator(new CouponTranslator());
 
// o => Coupon.IsValid(o.CouponCode, true)
//   -> jsv_coupon_strict(o.CouponCode)

Returning false passes the node on, so a translator handles only the shapes it recognises.

Your mapping, your runtime

Whatever a template or translator emits must exist at evaluation time. If you emit jsv_coupon_ok(...), defining jsv_coupon_ok in the JavaScript on the page is your responsibility. The library's contract is narrow: it emits text and ships its own stub - there is deliberately no companion-JS registration API, so how you bundle your own helpers is up to you.

Parameter rendering is separate

These registries cover types. How a rule's parameter renders - the form-validation reference('path') binding - is a different registry, compiler.Parameters, covered in Validating against a model object.
