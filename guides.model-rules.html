Validating against a model object

Brief

Write rules that span several fields of a model, bound to the form on the page.

Details

Most field checks need only that one field's value - a length, a digit, a postal-code format - and a single-value rule (the kind in the walkthrough) handles them. A model rule earns its place when a check has to see more than one field at once.

The classic example is a change-password form. Every rule is single-field - minimum length, contains a digit, and so on - except one: the new password and its confirmation must match. That check cannot be expressed against a single value; it has to read both fields, so it is written against the whole model.

A rule that spans two fields Hide

// the new password and its confirmation must be equal
Expression<Func<Account, bool>> rule = m => m.Password == m.Confirm;

The problem: where will JavaScript get the object and its properties from?

By default the compiler emits each member access verbatim: the rule above would become jsv_equal(m.Password, m.Confirm). That only works if the browser already has a JavaScript object m shaped exactly like the server model. On a form you do not - the values live in input fields, not in a mirror of your C# model.

What you want instead is for each access to resolve against the form: read Password and Confirm from the page.

The binding: reference()

The convention the library implements for you is a single host-provided JavaScript function, reference(): by default a model access resolves to a reference(...) call that reads the matching value from the page.

Hook

Provided by the page

reference('Path.To.Field')

Returns the current value of a model field, given its path. Your page implements it to read the matching input (or a data model bound to the form).

With that binding the password-match rule emits against the form instead of a server-shaped object:

The emitted rule Show

The emitted rule Hide

// rule:   m => m.Password == m.Confirm
// emits:  jsv_equal(reference('Password'), reference('Confirm'))

A field reached through a longer chain becomes a dotted path the same way - m.Address.PostalCode becomes reference('Address.PostalCode').

Map the model to the form, once for the whole application

You are not binding a particular model - you are declaring, once for the whole application, how any rule's model maps onto the client-side source (the form). The convention is the reference() hook: a model field resolves to the matching value on the page.

This is off by default: out of the box the compiler emits member access verbatim. You turn the reference() convention on with Parameters.MapReference, whose built-in rendering makes the bare parameter reference(), a member chain reference('Path.To.Field'), and array indexing jsv_index(...) - so m.Password, in any rule over any model, becomes reference('Password').

MapReference takes a Func<Type, bool> that picks which parameters it applies to; passing _ => true enables it for every parameter, of any type, in any rule. (You could filter by type, but a validation rule's one parameter is always the model, so there is nothing to filter.) It only ever touches the rule's own parameter, so a parameter introduced by a nested LINQ lambda (r => r.Length > 0) keeps its own name.

Because it is one unified convention, enable it once in a small application-level factory and run every rule through it - no rule repeats the setup:

Compile every rule through one factory Show

Compile every rule through one factory Hide

static string CompileRule(LambdaExpression rule)
{
    var compiler = new ExpressionCompiler(rule);
    compiler.Parameters.MapReference(_ => true);   // turn the reference() convention on for all
    return compiler.JavaScriptExpression;
}
 
// m => m.Password == m.Confirm
//   -> jsv_equal(reference('Password'), reference('Confirm'))

The host side

The page provides the hook the emitted code calls: reference(path), which walks the form by a dotted path and returns a field's current value. Write reference once for your app, alongside the runtime stub.

Serve each rule wrapped as a function, compiled through the factory above. Because the rule reads its inputs through reference(), the function takes no arguments:

Serving a compiled rule Show

Serving a compiled rule Hide

// GET /js/rules/{name} -> defines a function {name}() that calls reference()
app.MapGet("/js/rules/{name}", (string name) =>
{
    string expr = CompileRule(Rules.For(name));
    string script = $"function {name}() {{ return {expr}; }}";
    return Results.Text(script, "application/javascript");
});

On the page, implement reference against your form and call the rule:

Host page - reference() and running the rule Show

Host page - reference() and running the rule Hide

<form name="account">
    <input name="Password" type="password">
    <input name="Confirm"  type="password">
</form>
 
<!-- jsv_* runtime, then the rule (defines passwordsMatch()) -->
<script src="/js/expr-stub.js"></script>
<script src="/js/rules/passwordsMatch"></script>
<script>
    // written once for the app: resolve a model path against the form
    function reference(path) {
        return document.forms["account"].elements[path].value;
    }
 
    document.forms["account"].addEventListener("submit", function (e) {
        if (!passwordsMatch()) {
            e.preventDefault();
            showError("Passwords do not match.");
        }
    });
</script>

Referring to the model another way

MapReference emits reference('path'), which is all most pages need. When it isn't enough, Map lets you emit anything - and it takes two functions, because a rule can use the model in two shapes:

  • the whole model (the bare parameter) - for example when you pass it to a custom whole-entity check; and

  • a single field (a member chain rooted in the parameter), such as m.Password.

A custom whole-entity function shows both at once - register the function (server + client), then a rule that uses the model whole and by field:

The two shapes Show

The two shapes Hide

compiler.Methods.MapMethod(typeof(AccountRules), nameof(AccountRules.IsStrong),
                           "jsv_account_strong($0)");
compiler.Parameters.MapReference(_ => true);
 
Expression<Func<Account, bool>> rule =
    m => AccountRules.IsStrong(m) && m.Password == m.Confirm;
 
// ((jsv_account_strong(reference())) && (jsv_equal(reference('Password'), reference('Confirm'))))
//                      ^ whole model                  ^ a field

The first Map function renders the bare parameter (the whole model); the second renders a member or index chain rooted in it. MapReference simply fills both in for you (reference() and reference('path')).

More than one form on the page

With several forms on one page, reference('Password') is ambiguous. Pass the model's type as the form name so the page knows which form to read. The type is on the ParameterExpression, and ParameterAccessPath turns the chain into the dotted path:

A type-qualified reference Show

A type-qualified reference Hide

compiler.Parameters.Map(
    _ => true,
    p => $"reference('{p.Type.Name}')",
    (e, p) => $"reference('{p.Type.Name}', '{ExpressionCompiler.ParameterAccessPath(e)}')");
 
// m => m.Password == m.Confirm
//   -> jsv_equal(reference('Account', 'Password'), reference('Account', 'Confirm'))

On the page, the matching reference reads the named form:

The host-side reference(type, path) Show

The host-side reference(type, path) Hide

function reference(type, path) {
    return document.forms[type].elements[path].value;
}

An object instead of a lookup function

To skip the lookup function, refer to the model as an object the page exposes: the whole model is value and a field is value.Password. Nothing is replaced by the bare word value - it is the object the fields hang off:

Object access Show

Object access Hide

compiler.Parameters.Map(
    _ => true,
    p => "value",
    (e, p) => "value." + ExpressionCompiler.ParameterAccessPath(e));
 
// m => m.Password == m.Confirm
//   -> jsv_equal(value.Password, value.Confirm)

Whatever you emit, the only contract is that the page provides whatever the emitted code references.
