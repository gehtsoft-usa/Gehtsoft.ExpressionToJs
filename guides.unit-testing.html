Unit-testing your expressions

Brief

Prove the C# rule and its emitted JavaScript agree, by running both and comparing - the way this library tests itself.

Details

A validation rule is a boolean - it answers valid or not. Parity means the C# delegate and the emitted JavaScript return the same true/false for the same input. That is what a differential test checks: run both, compare the verdicts. It matters because C# and JavaScript diverge in ways that can flip that verdict - integer division, banker's rounding, date framing, == coercion, Unicode whitespace. The emitted jsv_* helpers exist to keep the two ends agreeing; a test is how you confirm they do for your rule.

The setup: an in-process JavaScript engine

You do not need a browser. Run the emitted JavaScript in-process with Jint, a JavaScript engine for .NET. The one rule that makes it work: load the runtime stub (ExpressionToJsStubAccessor) into the engine once, so the jsv_* helpers the emitted code calls are defined.

Because the result is always a boolean, the assertion is simple - no numeric coercion, just bool against bool:

using Jint;
using Gehtsoft.ExpressionToJs;
 
static void AssertRule<T>(Expression<Func<T, bool>> rule, T input)
{
    // 1. server side: compile and invoke the delegate
    bool expected = rule.Compile()(input);
 
    // 2. emit the JavaScript
    string js = new ExpressionCompiler(rule).JavaScriptExpression;
 
    // 3. run it in Jint with the stub loaded and the parameter bound
    var engine = new Engine();
    engine.Execute(ExpressionToJsStubAccessor.GetJsIncludesAsString());
    engine.SetValue(rule.Parameters[0].Name, input);
    bool actual = engine.Evaluate(js).AsBoolean();
 
    // 4. same verdict on both tiers?
    Assert.Equal(expected, actual);
}

// a range check, at the boundary
AssertRule<int>(age => age >= 18 && age <= 120, 18);
 
// "required" - tabs and other whitespace must fail, as in C#
AssertRule<string>(s => !string.IsNullOrWhiteSpace(s), "\t");
 
// integer division: C# says 11 / 2 == 5; the stub keeps JS agreeing
AssertRule<int>(n => n / 2 == 5, 11);

Binding the input

The verdict comparison is trivial; the care is in binding the input the same way the browser will:

  • Dates must be bound in the same frame as the compiler's DateTimeMode - bind a UTC date for a Utc rule, a local one for a Local rule. A frame mismatch in the test is the same bug it would be in production; see Dates and time zones.

  • Model rules need the parameter bound as an object Jint can read (SetValue("p", model)). If you compiled the rule for the form binding (reference()), define a reference function in the engine too, so the emitted lookups resolve.

What to test

Sampling round numbers hides exactly the bugs differential testing exists to catch. Aim your inputs at the places C# and JavaScript disagree:

  • Integer division and modulo with operands that do not divide evenly (11 / 2).

  • Math.Round on midpoints (0.5, 2.5) - banker's rounding.

  • Equality across types and the float-tolerance case (0.1 + 0.2 == 0.3).

  • null / empty inputs - strings, collections, nullable values.

  • Dates near midnight and across month/year boundaries, in the DateTimeMode you ship.

How this library tests itself

The same pattern guards the library. Its Complete/ differential suite checks each case three ways - the compiled C# delegate, the emitted JavaScript in Jint, and the two against each other - and that comparison is what keeps every server-side Functions helper in lock-step with its jsv_* twin in the stub. Pointing the same harness at your rules is the surest way to know your client and server agree.
