Validating on all tiers from one source of truth

Brief

Write a validation rule once in C#, run it on the server, and run the same rule in the browser.

Details

The goal of this library is one rule, two runtimes. You write a validation predicate once, as a C# lambda. The server runs it directly as a compiled delegate; the browser runs an equivalent JavaScript expression emitted from the same lambda. Because both ends come from a single definition, they cannot drift apart.

This walkthrough takes one rule from C# all the way into the page.

Step 1 - write the rule once

A rule is an ordinary Expression<Func<...>>, usually a boolean predicate over your model. Keep it an expression (not a statement lambda) so the compiler can walk it.

The rule Show

The rule Hide

using System.Linq.Expressions;
using Gehtsoft.ExpressionToJs;

Expression<Func<int, bool>> rule = x => x > 0 && x < 100;

Step 2 - run it on the server

On the server the rule is just a delegate. Compile it and invoke it like any Func. This is the authoritative check that cannot be bypassed.

Server side Show

Server side Hide

bool ok = rule.Compile()(42);   // true

Step 3 - emit the JavaScript

Hand the same lambda to ExpressionCompiler and read JavaScriptExpression. The result is computed once and cached, so configure the compiler (date mode, custom translators) before reading it.

Emitting the client code Show

Emitting the client code Hide

string js = new ExpressionCompiler(rule).JavaScriptExpression;
// js == "((jsv_greater(x, 0)) && (jsv_less(x, 100)))"

The emitted text is an expression, not a program. It calls small runtime helpers named jsv_* (jsv_greater, jsv_equal, jsv_match, ...). Those helpers exist so JavaScript reproduces C# semantics - integer division, banker's rounding, typed equality, date handling - rather than JavaScript's looser defaults. See Supported expression features for the full set.

Step 4 - ship the runtime stub to the page

The jsv_* helpers live in stub.js, shipped inside the assembly as an embedded resource. Retrieve it with ExpressionToJsStubAccessor.GetJsIncludesAsString and include it on the page once, before evaluating any compiled expression. It is the same for every expression, so cache it site-wide rather than per rule.

Getting the runtime Hide

string stub = ExpressionToJsStubAccessor.GetJsIncludesAsString();

There are two common ways to get that string to the browser.

Render it inline - write the stub straight into the page, for example from a Razor layout so every page that validates has it:

Inline in an ASP.NET Razor layout Show

Inline in an ASP.NET Razor layout Hide

@* _Layout.cshtml - emit the runtime once, e.g. in the page head *@
<script>
    @Html.Raw(ExpressionToJsStubAccessor.GetJsIncludesAsString())
</script>

Serve it as a static script - the stub is just a set of jsv_* function definitions, so you can expose it at a stable URL and reference it with a normal <script> tag (no wrapping needed - it defines the helpers as globals). It never changes between requests, so the browser caches it like any other static script - the better option for a busy site:

A REST endpoint that returns the stub Hide

// Program.cs - the stub is constant, so this response is highly cacheable
app.MapGet("/js/expr-stub.js", () =>
    Results.Text(ExpressionToJsStubAccessor.GetJsIncludesAsString(), "application/javascript"));

Step 5 - evaluate in the browser

If you rendered the rule inline, bind the variables the expression references (here x) and evaluate the emitted string. It returns the same boolean the server computed:

Evaluating inline Show

Evaluating inline Hide

// stub.js already loaded once on the page
var x = 42;
var ok = ((jsv_greater(x, 0)) && (jsv_less(x, 100)));   // true

In a script-tag setup you fetch each rule from the server as well. A bare expression is not callable on its own, so wrap it in a named function - the name fixed, or taken from the request - and return that:

A REST endpoint that returns the rule as a function Show

A REST endpoint that returns the rule as a function Hide

// GET /js/rules/{name} -> a script defining a function called {name}
app.MapGet("/js/rules/{name}", (string name) =>
{
    string expr = new ExpressionCompiler(Rules.For(name)).JavaScriptExpression;
    string script = $"function {name}(x) {{ return {expr}; }}";
    return Results.Text(script, "application/javascript");
});

Load the stub once, the rule script alongside it, and call the function like any other:

Loading the rule and calling it Show

Loading the rule and calling it Hide

<!-- the jsv_* runtime, once -->
<script src="/js/expr-stub.js"></script>
<!-- defines function ageOk(x) -->
<script src="/js/rules/ageOk"></script>
<script>
    if (!ageOk(document.getElementById("age").value))
        showError("Enter a value between 1 and 99.");
</script>

Beyond a single value

The rule above validates one base-type value bound to a bare variable. Some rules instead have to see several fields at once - for example, that a password and its confirmation match. Those are written against the whole model and bound to the form on the page, which is a little extra setup covered in its own guide: Validating against a model object.
